1. Executive summary

ABI provides per-rental car insurance for peer-to-peer / car-share platforms. Hosts buy an off-rental "Period X" policy for each vehicle; on top of that, the platform can programmatically issue "Period Z" on-rental coverage every time a renter books that vehicle.

Amgad (one of our tenants) is moving to ABI for his fleet and asked whether Drive247 will integrate. Neema confirmed during the call that an integration is desirable but not blocking — Amgad can operate through ABI's portal manually until we build it.

Recommendation: defer the build. Bonzah already covers the renter-facing upsell; ABI's Period X enrollment API does not yet exist for car-share hosts; and ABI's carrier-name-on-card rule prevents the full white-label experience Amgad wants. Park as a Phase 2 product feature and revisit when 2–3 tenants ask for it or ABI ships the host-onboarding API.

2. What ABI Insurance is

One line: ABI provides per-rental car insurance. Every time a customer rents a vehicle on Drive247, ABI insures that specific rental for the days it runs. The renter receives an insurance ID card with their name on it, valid only for that booking.

It is purpose-built for peer-to-peer car sharing and rental marketplaces. Hosts sign up once, then issue per-rental policies on top of their fleet through API calls from the platform.

3. The two-period concept

A rental vehicle has two distinct insurance phases:

PeriodWhen it appliesWho drivesWho pays
Period X
(off-rental)		 Car at host's lot, parked at home, host driving to wash/fuel Host (owner) Host — bought once, billed monthly
Period Z
(on-rental)		 Car is rented to a customer Renter Per-rental fee — passed through by platform
Critical rule: a vehicle can only get Period Z if it already has an active Period X policy. Period X is the gating prerequisite for everything else.

4. Concrete example — Amgad's flow

Setup (one-time, per vehicle)

  • Amgad owns 5 cars and signs up at portal.abiweb.com.
  • He buys Period X for each car (manual today — no API for new-host enrollment).
  • All 5 cars are now insured at rest and eligible for on-rental coverage.

A booking happens

  1. Sarah opens Amgad's Drive247 site, books his Tesla Model 3 for 25 May → 28 May (3 days).
  2. Drive247 calls ABI: "Is VIN 5YJ3E1EA1JF… eligible?"
    → ABI: ✅ yes, Period X is active.
  3. Drive247 sends Sarah's details: Add Renter (name, DOB, license #, license state, address, phone, email) → ABI returns RENTERID.
  4. Drive247 creates the rental period: Create Rental (VIN + renter ID + start + end + state + timezone + usage type) → ABI returns RENTALID.
  5. Drive247 fetches the card: Get ID Card → ABI returns a PDF/URL showing Sarah as insured driver, VIN, dates, carrier, policy limits.
  6. Drive247 emails the card to Sarah and attaches it to the rental record. She is legally covered to drive that car for 3 days.

Rental ends

  1. Sarah returns the car on 28 May. Drive247 calls End Rental → coverage stops.
  2. ABI bills Amgad (or Drive247) for 3 days × daily rate for that Tesla.

5. ABI vs Bonzah — they solve different problems

BonzahABI
What it isRenter damage waiver (optional add-on)Primary commercial insurance on the vehicle
Who buysThe renter at checkoutHost (Period X) + platform issues Period Z per rental
What it coversRenter's damage / excess liabilityLegal liability + comp / collision while in commercial rental use
Without itRenter pays out-of-pocket for damageHost has no legitimate commercial coverage → personal policy denies claims → host is fully exposed
Optional?Yes — upsellNo — required for legal commercial operation

Bonzah is a renter-facing upsell. ABI is the carrier of record for the vehicle being rented. A real fleet operation typically has both.

6. Meeting takeaways (21 May 2026)

Confirmed facts

  • ABI's carrier name must legally appear on every ID card. Drive247 cannot fully white-label.
  • Period X must be purchased by the host first; without it, vehicles cannot be made eligible for Period Z.
  • ID card displays the renter's name and the vehicle VIN (Chris confirmed verbally; a "renter-name-on-card" defect was discussed and a ticket has been filed).
  • No separate sandbox URL — staging and production share https://api.abiweb.com; only credentials differ.
  • Hosts must already be ABI customers; brand-new host enrollment is not yet API-driven.
  • Volume discount tiers: 1–3 vehicles, 4–9, 10+ (max discount at 10). Pricing covers liability with optional comprehensive + collision.

Amgad's concerns

  • Wants to hide ABI from his hosts entirely — branding risk if hosts learn the carrier name.
  • Wants to start with his own 5–9 cars before introducing to third-party hosts.
  • Asked whether coverage is full or liability-only — answer: configurable (add comp/collision on signup).

Neema's framing

"You can insure your vehicles now and continue your business. When the build is finished you'll be able to do it from within the system — but it's not a blocker."

7. API documentation review

What is documented

  • Auth: Basic Auth (portal credentials, base64-encoded) + a 2FA token in a Token header for "required users". Endpoints exist to request and verify the 2FA code.
  • Base URL: https://api.abiweb.com (staging and production share the URL).
  • Core Period Z endpoints:
    • GET /period-z/eligibility/vin/{VIN}/ — Vehicle Eligibility Check
    • POST /customer/{CN}/period-zero/renter/ — Add Renter
    • POST /customer/{CN}/policy/{PN}/period-zero/rental/ — Create Rental Period
    • POST /customer/{CN}/policy/{PN}/period-zero/rental/{VIN}/end/ — End Rental
    • DELETE /customer/{CN}/policy/{PN}/period-zero/rental/{VIN}/ — Cancel Rental
    • GET /customer/{CN}/policy/{PN}/period-zero/rental/{RENTALID}/idcard/ — Proof of Insurance
    • GET /period-z/states-allowed/ — Allowed States
    • GET /customer/{CN}/billing/vins/?startDate=&endDate= — Cost by VIN
  • Renter object fields: firstname, lastname, license state, license number, phone, email, DOB, address, city, state, ZIP.
  • Rental object fields: VIN, RENTERID, STATE, TIMEZONE, STARTTIME, ENDTIME, USAGETYPE (Personal or Rideshare), optional EXTERNALREF.
  • USAGETYPE controls renter name on card: Rideshare shows renter name (Uber/Lyft style); Personal omits it (Turo style). The "bug" mentioned in the call is likely a configuration mismatch.

What is NOT documented (gaps)

  • 200 / 400 response schemas are referenced but not fully described.
  • ID card payload format (PDF binary vs URL vs base64).
  • 2FA token lifetime / TTL.
  • Cancellation window and refund / prorating rules.
  • Per-VIN daily rate structure or pricing breakdown.
  • Webhook / event subscription support — none documented; appears to be poll-only.

8. Period X enrollment — what's actually possible today

Chris's verbal statement during the call ("we don't have this endpoint set up yet") was correct but oversimplified. The nuanced picture:

CapabilityEndpointSupported?
Enrol a brand-new car-share host into Period X—❌ No — must use portal.abiweb.com
Enrol a new TNC driver (Uber Black)POST /quote/⚠️ Yes, but only for TYPE: uber-black
Get rating / pricing quotePOST /quote/rating/✅ Pricing only, not enrolment
Add a new vehicle to an existing Period X policyPOST /customer/{CN}/policy/{PN}/vehicles/✅ Yes
List existing policies for a customerGET /customer/{CN}/policies/✅ Yes
Process changes (mid-term edits)POST /customer/{CN}/policy/{PN}/changes/✅ Yes

So the realistic flow today is:

  1. One-time per host: manual signup on portal.abiweb.com. Drive247 records their customer + policy numbers once.
  2. Everything after that: adding a new car to their fleet, eligibility check, creating rentals, issuing cards, ending rentals, billing reconciliation — all API-driven from Drive247.

That's closer to 95% automated with one manual step at host signup, not "half-automated" as initially framed.

9. Requirements — Phase 1 / Phase 2 / Phase 3

PHASE 1 To get started (sandbox build)

  1. Test portal credentials — explicitly assigned by ABI account manager.
  2. 2FA setup for the test account + token TTL (not documented).
  3. Our test customer number (path param on every Period Z endpoint).
  4. Our test Period Z master policy number.
  5. Sample response payloads for Eligibility / Create Rental / Get ID Card — plus confirmation of ID card format (PDF binary vs URL vs base64).
  6. At least one test VIN with active Period X coverage on our account — without it, no test call can succeed because eligibility check will fail.

PHASE 2 To go live (production cutover)

  1. Production credentials + live customer / policy numbers.
  2. USAGETYPE decision (Personal vs Rideshare) — controls whether renter name appears on card; resolves the "bug" Chris mentioned.
  3. Cancellation rules — time window, refund policy, proration on early End Rental.
  4. Pricing model — daily rate structure, flat vs by class/state, billing cadence (per call vs monthly invoice).
  5. List of supported states for our policy.
  6. Vehicle eligibility rules — age, value cap, salvage title, mileage.
  7. Driver eligibility rules — min age, license type, MVR check requirements.
  8. Period X enrolment workflow for hosts (manual portal vs co-brand option).
  9. Coverage details for the ID card — limits (BI/PD), deductibles, liability vs full coverage.
  10. Sample ID card so we know what renters receive.
  11. Required legal disclosures at booking checkout.
  12. Producer / broker license question for our role.

PHASE 3 Eventually (post-launch / scale)

  1. Period X (host onboarding) API when ABI ships it — true white-label.
  2. Webhooks for policy / rental / claim events.
  3. Claims workflow integration.
  4. Reconciliation endpoints — list active rentals, audit reports per host.
  5. Multi-tenant ABI accounts — per-tenant customer/policy pairs (like Bonzah / Stripe Connect).
  6. Volume discount tiers for Period Z.
  7. Co-branded ID cards.
  8. Geographic expansion as ABI adds states.
  9. Pre-booking renter pre-qualification.
  10. Reporting / analytics dashboards.
  11. GPS / telematics feed (Report Location endpoint exists).
  12. Data retention & PII processing agreement.

10. Phase 1 — the 4 certain asks

After verifying against the docs, only these four are 100% certain blockers. Everything else can be phrased as a question rather than a firm requirement.

  1. Test portal credentials — quoted directly from docs: "Your account manager will setup and provide you with test credentials on an as needed basis."
  2. Our test customer number — every Period Z endpoint has {CUSTOMERNUMBER} in its path.
  3. Our test Period Z master policy number — every rental endpoint has {POLICYNUMBER} in its path.
  4. A test VIN with active Period X coverage on our test account — the eligibility check gates the entire flow; without a Period X-covered vehicle, no rental can be created in test.

11. Decisions & recommendation

Decision: defer the build

Bonzah already serves the renter-facing upsell. ABI fills a different need (primary commercial coverage on the vehicle), but its delivery does not yet support the white-label experience Amgad is asking for, and the host enrolment path is still manual. Amgad can operate through ABI's portal today without any Drive247 changes.

Triggers to revisit

  • A second or third tenant requests ABI integration (signals it's a product, not a one-off).
  • ABI ships the Period X enrolment API for car-share hosts (unlocks true white-label).
  • Amgad scales past ~15–20 cars and the manual portal workflow becomes painful enough that he asks again.
  • Drive247 wants to offer commercial fleet insurance as a standalone product line.

If we do build

The integration shape is well understood — it follows the Bonzah pattern (call API, pass renter + vehicle + dates, receive policy + ID card). Roughly:

  • Settings → Integrations tab with an ABI connector (test/live toggle, customer + policy number per tenant).
  • Per-vehicle eligibility status cached and refreshed on demand.
  • Hook into the rental creation flow: on confirmation, call Add Renter → Create Rental → Get ID Card → attach to the rental record + email to renter.
  • Hook into rental completion: call End Rental.
  • Hook into rental cancellation: call Cancel Rental.
  • Mirrors the existing Bonzah and BoldSign patterns; effort estimate roughly comparable.

12. Open questions for Neema / ABI

  1. Is the 2FA token requirement applied to API service users, or only to portal-login users?
  2. What is the 2FA token lifetime — once obtained, can it be cached?
  3. What format does the ID card endpoint return (PDF binary, URL, base64)?
  4. For Drive247's use case, should USAGETYPE be Personal or Rideshare? Renter name on card?
  5. What is the cancellation window, and what proration/refund applies on early End Rental?
  6. Pricing model — flat daily rate per VIN, or rated by class/state/coverage?
  7. Is there an ETA for a car-share host enrolment API (beyond the existing uber-black-only path)?
  8. What is the supported state list for our policy?
  9. Are webhooks on the roadmap, or are we expected to poll?
  10. Does Drive247 need to be set up as the producer / retail broker of record before we can issue cards?
